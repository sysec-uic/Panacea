## 2. Root-cause analysis

The bug is a stack-use-after-return in mruby's bigint scratch-pool infrastructure.

mruby-bigint uses a performance optimization: a small stack-local scratch pool (`pool_storage`, declared by `MPZ_CTX_INIT` in `mrb_bint_rshift`) avoids heap allocation for intermediate `mpz_t` values. Temporary values are allocated from this pool via `mpz_init_temp`. The pool is valid only for the lifetime of that stack frame.

The sequence that triggers the crash:

1. `mrb_bint_rshift` initialises a `MPZ_CTX_INIT` context with `pool_storage` on its stack, then calls `mpz_div_2exp`.
2. `mpz_div_2exp` calls `mpz_init_temp` to allocate its working variable `y` from that pool.
3. When the shift amount is a whole number of limbs (`bs == 0`), `mpz_div_2exp` calls `mpz_move(z, &y)` — so the **output** `z` now points into stack pool memory.
4. `bint_new(ctx, &z)` is called to wrap `z` into a persistent GC-allocated bigint object. In the heap branch it calls `mpz_move(ctx, &b->as.heap, &z)`, which copies the pointer as-is — the GC object now holds a pointer into `mrb_bint_rshift`'s stack.
5. `mrb_bint_rshift` returns and its stack frame (including `pool_storage`) is destroyed.
6. Later, `[1].flatten(bigint)` triggers `ary_flatten`, which calls `mrb_ensure_int_type` on the dangling bigint → `mrb_bint_as_int` → `mpz_get_int` reads from the freed stack region → stack-use-after-return.

The escape boundary is `bint_new`: that is the exact moment an ephemeral `mpz_t` becomes a persistent object, and the only point where pool memory must be promoted to the heap.

## 3. The fix

In `bint_new`, when taking the heap branch, detect whether `x` is backed by pool memory using `is_pool_memory`. If it is, copy the limbs to a real heap allocation via `mpz_init_heap` + `memcpy` instead of blindly calling `mpz_move` (which would just steal the stack pointer):

```c
else {
  RBIGINT_SET_HEAP(b);
#if MRB_BIGINT_POOL_SIZE > 0
  if (MPZ_HAS_POOL(ctx) && is_pool_memory(x, MPZ_POOL(ctx))) {
    /* x is backed by the scratch pool, which lives on the stack and is
       released when the current operation returns.  A persistent bigint
       object must not retain such a pointer, so copy the limbs to the heap
       instead of moving the (soon to be dangling) pool pointer. */
    mpz_init_heap(ctx, &b->as.heap, x->sz);
    memcpy(b->as.heap.p, x->p, x->sz*sizeof(mp_limb));
    b->as.heap.sn = x->sn;
    mpz_clear(ctx, x);
  }
  else
#endif
  {
    mpz_move(ctx, &b->as.heap, x);
  }
}
```

The guard is conditional on `MRB_BIGINT_POOL_SIZE > 0` so that pool-disabled builds are unaffected. The `mpz_clear` on the pool-backed `x` is safe because `mpz_clear` on pool memory is a no-op (the pool is freed all at once on scope exit).
