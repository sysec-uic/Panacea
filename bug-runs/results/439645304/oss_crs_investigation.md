## 2. Root-cause analysis

The bug is a missing GC write barrier in mruby's `Set` implementation. mruby uses a tri-color incremental GC (white/gray/black). The invariant is: a black object must never reference a white object — if it does, the white object will be swept even though it's still live.

The sequence that triggers the crash:

1. A `Set` object is marked black by the incremental GC sweep.
2. A new object is inserted into the Set via `set_add` → `kset_put` — but **no write barrier is fired**.
3. The GC continues and sweeps the newly-inserted object (it's white and unreachable from the gray worklist).
4. Later, `set_add` triggers a hash table resize (`kh_resize_set_val` → `kh__rebuild_set_val`), which iterates all keys and rehashes them.
5. Rehashing calls `kset_hash_value` → `mrb_obj_hash_code` → `mrb_type` on the freed object → heap-use-after-free.

The freed-by stack confirms the GC sweep fires during `mrb_obj_alloc` for a new closure, triggered by `mrb_obj_hash_code` calling a Ruby `hash` method mid-insertion. The write barrier is present in mruby's core `hash.c` and `array.c` implementations but was never added to `mruby-set`.

## 3. The fix

Add a `set_put_value` wrapper that calls `mrb_field_write_barrier_value` after every insert into the underlying `kset_t`. The owning `RSet` is recovered from the embedded `kset_t` pointer via `offsetof`, mirroring the pattern used in `hash.c` / `array.c`:

```c
static void
set_put_value(mrb_state *mrb, kset_t *set, mrb_value key, int *ret)
{
  if (ret) {
    kset_put2(mrb, set, key, ret);
  }
  else {
    kset_put(mrb, set, key);
  }
  struct RSet *s = (struct RSet*)((char*)set - offsetof(struct RSet, set));
  mrb_field_write_barrier_value(mrb, (struct RBasic*)s, key);
}
```

All eight `kset_put` / `kset_put2` call sites in `set.c` are replaced with `set_put_value`:
`kset_copy_merge`, `set_add`, `set_add_p`, `set_core_intersection`, `set_core_xor` (×2), `set_add_all`, `set_flatten_recursive`, `set_s_create`.
