## 2. Root-cause analysis

The agent reproduced the crash and traced how pool memory flows into a persistent object. The POC involved rational literals with huge exponents (near `UINT64_MAX`). After examining `udiv`, `mpz_init_temp`, and `mpz_move`, it identified the escape point:

> "In `udiv` (and similar functions), `mpz_init_temp` can allocate from a stack-based pool (`pool_storage`). The result is then transferred to the caller's output via `mpz_move`, which just copies the pointer. So the output `mpz` ends up pointing into the stack pool. When that output escapes into a persistent `RBigint` via `bint_new` → `mpz_move`, the bigint's data points to a stack frame that's later destroyed → stack-use-after-return."

The agent confirmed pool size is 512 (enabled) and concluded `mpz_move` is the right universal fix point — it must never transfer ownership of transient pool memory to a persistent destination.

Same pool-aliasing class as bug 439237851, but the escape point is `mpz_move` rather than `bint_new`.

## 3. The fix

Intercept inside `mpz_move`: when the source `x->p` is pool-backed, copy the limbs to the
heap before transferring ownership, then null out `x` to prevent double-free if the pool
scope is later cleared:

```c
static void
mpz_move(mpz_ctx_t *ctx, mpz_t *y, mpz_t *x)
{
  mpz_clear(ctx, y);
#if MRB_BIGINT_POOL_SIZE > 0
  if (x->p && MPZ_HAS_POOL(ctx) && is_pool_memory(x, MPZ_POOL(ctx))) {
    /* x->p lives in the stack pool — copy to heap before transferring ownership */
    y->sn = x->sn;
    y->sz = x->sz;
    if (x->sz > 0) {
      y->p = (mp_limb*)mrb_malloc(MPZ_MRB(ctx), x->sz * sizeof(mp_limb));
      memcpy(y->p, x->p, x->sz * sizeof(mp_limb));
    }
    else {
      y->p = NULL;
    }
    x->p = NULL; x->sn = 0; x->sz = 0;
    return;
  }
#endif
  y->sn = x->sn;
  y->sz = x->sz;
  y->p = x->p;
  ...
}
```

The guard compiles out entirely when `MRB_BIGINT_POOL_SIZE == 0`. See `oss_crs_patch.diff`.
