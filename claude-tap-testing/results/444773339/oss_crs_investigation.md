## 1. Reproduce

The PoC is a Ruby script using large `**` (power) operations. `33333333333333533375` exceeds
`INT64_MAX` so it becomes a BigInt; `** 3331` triggers `mrb_bint_pow`, which internally calls
`mpz_add` → `uadd`. Running the unpatched fuzzer directly against the PoC:


## 2. Root-cause analysis

The POC initially timed out through the run-pov sidecar (30s), so the agent worked backwards from the Ruby input: `33333333333333533375` exceeds `INT64_MAX` so it becomes a BigInt, and `** 3331` calls `mrb_bint_pow`. Running the fuzzer binary directly confirmed a heap-buffer-overflow crash in 0.4s.

After symbolizing the stack the agent pinpointed `uadd` (bigint.c:356) and identified the bug:

> "`uadd(z, x, y)` requires `y->sz >= x->sz`: its first loops iterate `x->sz` limbs reading both `x->p[i]` and `y->p[i]`, then handle remaining limbs from larger operand `y`, and write the final carry to `z->p[y->sz]`. But `mpz_add` calls `uadd(&z, x, y)` without ordering by size. When `x->sz > y->sz`, the first loop reads `y->p[i]` past `y`'s allocation — exactly the 4-byte read 0 bytes after the 13-limb (52-byte) region. The upstream `uadd` guards this by swapping. This refactored '4x unrolled' version dropped the swap."

```c
for (i = 0; i + 4 <= x->sz; i += 4) {
    c += (mp_dbl_limb)y->p[i] + (mp_dbl_limb)x->p[i];   // reads y->p[i]
    ...
}
for (; i < x->sz; i++) {
    c += (mp_dbl_limb)y->p[i] + (mp_dbl_limb)x->p[i];   // reads y->p[i] out of bounds when y->sz < x->sz
    ...
}
```

Note: the run-pov sidecar timed out at 30s on both base and patched builds in this environment (even a trivial `1+1` input timed out). Ground-truth validation used the fuzzer binary directly.

## 3. The fix

Restore the swap at the top of `uadd` so `y` is always the operand with at least as
many limbs as `x` (addition is commutative, so the swap is semantically correct):

```c
static void
uadd(mpz_t *z, mpz_t *x, mpz_t *y)
{
  if (y->sz < x->sz) {
    mpz_t *t = x; x = y; y = t;
  }
  ...
}
```

See `oss_crs_patch.diff`.
