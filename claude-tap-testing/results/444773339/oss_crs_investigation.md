## 1. Reproduce

The PoC is a Ruby script using large `**` (power) operations. `33333333333333533375` exceeds
`INT64_MAX` so it becomes a BigInt; `** 3331` triggers `mrb_bint_pow`, which internally calls
`mpz_add` → `uadd`. Running the unpatched fuzzer directly against the PoC:

```
==ERROR: AddressSanitizer: heap-buffer-overflow on address ... READ of size 4
  0 bytes after a 52-byte region
    #0 uadd        mrbgems/mruby-bigint/core/bigint.c:356
    #1 mpz_add     mrbgems/mruby-bigint/core/bigint.c
    #2 mrb_bint_add ...
SUMMARY: heap-buffer-overflow ... in uadd
```

## 2. Root-cause analysis

`uadd(z, x, y)` adds two unsigned multi-limb integers. It requires that `y->sz >= x->sz`:
its first block of loops iterates `i < x->sz` reading **both** `x->p[i]` and `y->p[i]`,
then a second block walks the *remaining* limbs of `y` past that point:

```c
for (i = 0; i + 4 <= x->sz; i += 4) {
    c += (mp_dbl_limb)y->p[i] + (mp_dbl_limb)x->p[i];   // reads y->p[i]
    ...
}
for (; i < x->sz; i++) {
    c += (mp_dbl_limb)y->p[i] + (mp_dbl_limb)x->p[i];   // reads y->p[i]
    ...
}
/* then continues: for (; i < y->sz; i++) ... z->p[y->sz] = ... */
```

`mpz_add` calls `uadd(&z, x, y)` **without ordering the operands by size**. When
`x->sz > y->sz`, the first loop reads `y->p[i]` for `i >= y->sz` — 4 bytes past
the end of `y`'s 13-limb (52-byte) allocation — and later writes a carry to
`z->p[y->sz]` which is also out-of-bounds.

The upstream `uadd` guards this with a size-ordering swap at entry. This refactored
"4x unrolled" version dropped that swap.

## 3. The fix

Restore the swap at the top of `uadd` so `y` is always the operand with at least as
many limbs as `x` (addition is commutative, so the swap is semantically correct):

```c
static void
uadd(mpz_t *z, mpz_t *x, mpz_t *y)
{
  if (y->sz < x->sz) {
    mpz_t *t = x; x = y; y = t;
  }
  ...
}
```

See `oss_crs_patch.diff`.
