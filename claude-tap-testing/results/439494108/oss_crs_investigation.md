## 2. Root-cause analysis

The agent reproduced the crash and traced the escape path through `udiv` and `mpz_mdiv`, noting a comment in the code that made the intent clear but the implementation wrong:

> "The pattern is clear. The intent (see comment 'Copy results to avoid pool/heap mixing') is that `mpz_move` produces results that survive `pool_restore` and stack unwinding. But `mpz_move` merely transfers the pointer — when the source is pool (stack) memory, the destination ends up aliasing soon-to-be-reclaimed stack storage. In `udiv`, the quotient `q` is allocated from the pool via `mpz_init_temp`, then `mpz_move(ctx, qq, &q)` aliases it into the caller's output, which then escapes into a persistent `RBigint` via `bint_new` → stack-use-after-return."

The pool pointer escapes `mrb_bint_reduce` through this `mpz_move` call. When `mrb_bint_reduce` returns, its stack frame (including `pool_storage`) is reclaimed. The caller chain (`rat_sub_b` → `bint_mul` → `mpz_mul` → `mpz_mul_basic` → `limb_addmul_1`) subsequently reads the dangling pointer.

Same pool-aliasing class as 439237851 and 439279102, triggered via a different call path (`rat_sub_b` rational subtraction rather than `mrb_bint_div`/`mrb_bint_as_float`).

## 3. The fix

Intercept inside `mpz_move`: when the source `x->p` is pool-backed, copy the limbs to the
heap before transferring ownership, then null out `x`:

```c
static void
mpz_move(mpz_ctx_t *ctx, mpz_t *y, mpz_t *x)
{
  mpz_clear(ctx, y);
#if MRB_BIGINT_POOL_SIZE > 0
  if (MPZ_HAS_POOL(ctx) && is_pool_memory(x, MPZ_POOL(ctx))) {
    y->sn = x->sn;
    y->sz = x->sz;
    if (x->sz > 0) {
      y->p = (mp_limb*)mrb_malloc(MPZ_MRB(ctx), x->sz * sizeof(mp_limb));
      memcpy(y->p, x->p, x->sz * sizeof(mp_limb));
    }
    else {
      y->p = NULL;
    }
    x->p = NULL; x->sn = 0; x->sz = 0;
    return;
  }
#endif
  y->sn = x->sn;
  y->sz = x->sz;
  y->p = x->p;
  ...
}
```

The guard compiles out entirely when `MRB_BIGINT_POOL_SIZE == 0`. See `oss_crs_patch.diff`.
