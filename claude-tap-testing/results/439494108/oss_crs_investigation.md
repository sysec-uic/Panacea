## 2. Root-cause analysis

Same pool-aliasing class as bugs 439237851 and 439279102, but triggered via a different
call path.

`mrb_bint_reduce` (bigint.c:3673) declares a stack pool (`pool_storage`) and backs temporaries
`a` and `b` from it. Inside the reduction, `udiv` allocates its quotient `q` from the pool via
`mpz_init_temp`, then calls `mpz_move(ctx, qq, &q)` to transfer the result to the caller's
output. A comment in the code reads "Copy results to avoid pool/heap mixing", but `mpz_move`
only copies the raw pointer тАФ when the source is pool-backed, the destination ends up aliasing
soon-to-be-reclaimed stack storage.

The pool pointer escapes `mrb_bint_reduce` through this `mpz_move` call. When
`mrb_bint_reduce` returns, its stack frame (including `pool_storage`) is reclaimed. The
caller chain (`rat_sub_b` тЖТ `bint_mul` тЖТ `mpz_mul` тЖТ `mpz_mul_basic` тЖТ `limb_addmul_1`)
subsequently reads the dangling pointer тАФ stack-use-after-return at offset 160 inside
`pool_storage`.

## 3. The fix

Intercept inside `mpz_move`: when the source `x->p` is pool-backed, copy the limbs to the
heap before transferring ownership, then null out `x`:

```c
static void
mpz_move(mpz_ctx_t *ctx, mpz_t *y, mpz_t *x)
{
  mpz_clear(ctx, y);
#if MRB_BIGINT_POOL_SIZE > 0
  if (MPZ_HAS_POOL(ctx) && is_pool_memory(x, MPZ_POOL(ctx))) {
    y->sn = x->sn;
    y->sz = x->sz;
    if (x->sz > 0) {
      y->p = (mp_limb*)mrb_malloc(MPZ_MRB(ctx), x->sz * sizeof(mp_limb));
      memcpy(y->p, x->p, x->sz * sizeof(mp_limb));
    }
    else {
      y->p = NULL;
    }
    x->p = NULL; x->sn = 0; x->sz = 0;
    return;
  }
#endif
  y->sn = x->sn;
  y->sz = x->sz;
  y->p = x->p;
  ...
}
```

The guard compiles out entirely when `MRB_BIGINT_POOL_SIZE == 0`. See `oss_crs_patch.diff`.
