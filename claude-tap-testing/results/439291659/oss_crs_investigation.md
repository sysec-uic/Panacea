## 2. Root-cause analysis

Same pool-aliasing class as bug 439237851, but detected by MSan at GC time rather than ASan
at read time.

`bint_new` (bigint.c:2792) promotes a transient `mpz_t` into a GC-managed `RBigint`. For
large values (heap path) it calls `mpz_move(&b->as.heap, x)`, which copies `x->p` directly
into the persistent object. If `x->p` points into the stack-backed scratch pool (declared via
`MPZ_CTX_INIT` in the calling operation), the `RBigint` now holds a pool pointer instead of a
heap pointer.

`bint_mul` calls `mrb_as_bint` → `bint_new` with a pool-backed operand. The returned
`RBigint` escapes into the GC. When GC later sweeps the object it calls
`mrb_gc_free_bint` → `mpz_clear` → `free(b->as.heap.p)`. MSan's allocator checks that the
pointer is page-aligned heap memory; a stack pool address fails this check → CHECK failure.

The difference from 439237851 (ASan stack-use-after-return) is purely in sanitizer: ASan
catches the *read* of dead stack memory, MSan catches the *free* of non-heap memory.

## 3. The fix

In `bint_new`'s heap path, detect pool-backed source memory and copy limbs to the heap before
building the persistent object. Then call `mpz_clear` on the source to reset it without
attempting to free the pool storage:

```c
  else {
    RBIGINT_SET_HEAP(b);
#if MRB_BIGINT_POOL_SIZE > 0
    if (MPZ_HAS_POOL(ctx) && is_pool_memory(x, MPZ_POOL(ctx))) {
      size_t sz = x->sz;
      mp_limb *heap_p = (mp_limb*)mrb_malloc(MPZ_MRB(ctx), sz * sizeof(mp_limb));
      memcpy(heap_p, x->p, sz * sizeof(mp_limb));
      b->as.heap.p = heap_p;
      b->as.heap.sz = sz;
      b->as.heap.sn = x->sn;
      mpz_clear(ctx, x);  /* reset pool source without calling free */
    }
    else
#endif
    mpz_move(ctx, &b->as.heap, x);
  }
```

The guard compiles out entirely when `MRB_BIGINT_POOL_SIZE == 0`. See `oss_crs_patch.diff`.
