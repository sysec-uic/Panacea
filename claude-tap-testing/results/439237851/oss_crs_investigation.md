## 2. Root-cause analysis

The agent reproduced the crash immediately: stack-use-after-return in `mrb_bint_mod`/`mrb_bint_div` in `bigint.c`. It then examined `bint_new` and traced how temporaries get pool memory.

The finding: in `bint_new`, the heap branch uses `mpz_move` which transfers the `mpz_t`'s data pointer directly into the long-lived heap-allocated `RBigint`. When that data lives in the stack-allocated pool (`pool_storage`), the `RBigint` ends up holding a dangling pointer into a stack frame that's freed when the function returns — causing the stack-use-after-return when a later operation reads it.

Each bignum operation declares a stack pool via `MPZ_CTX_INIT`:

```c
mp_limb pool_storage[MRB_BIGINT_POOL_SIZE];   // lives on the current stack frame
```

`mrb_bint_div` and `mrb_bint_mod` both end with `bint_new(ctx, &z)` where `z` is computed through pool-backed temporaries. For a large enough result the heap path runs and `mpz_move` copies the raw pool pointer into the persistent `RBigint`. When `div` returns, that stack memory is gone — the subsequent `mod` reads it → stack-use-after-return in `uzero_p`.

## 3. The fix

At the escape boundary (`bint_new`), detect when the source pointer lives in the
pool and copy to heap instead of aliasing:

```c
  else {
    RBIGINT_SET_HEAP(b);
#if MRB_BIGINT_POOL_SIZE > 0
    if (MPZ_HAS_POOL(ctx) && is_pool_memory(x, MPZ_POOL(ctx))) {
      /* x->p lives in the stack pool, which is reclaimed when the current
         operation returns. Copy limbs to the heap before moving. */
      mp_limb *heap_p = (mp_limb*)mrb_malloc(MPZ_MRB(ctx), x->sz * sizeof(mp_limb));
      memcpy(heap_p, x->p, x->sz * sizeof(mp_limb));
      x->p = heap_p;
    }
#endif
    mpz_move(ctx, &b->as.heap, x);
  }
```

The guard compiles out entirely when `MRB_BIGINT_POOL_SIZE == 0`. See `oss_crs_patch.diff`.
